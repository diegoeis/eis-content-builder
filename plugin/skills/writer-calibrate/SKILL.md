# SKILL

> **Synopsis.** Learn from a correction. Mode A: compare plugin draft vs your rewrite → extracts voice diff → proposes rule deltas. Mode C: you describe what was off → targeted follow-ups → translates to concrete rules. Never touches `voice-fingerprint.md`. Always shows diff before writing.


# Writer Calibrate

Learn from user corrections. Update the style rules with real evidence. Never guess what the user meant.

## Invariant rules

Inherit from `{workspace}/CLAUDE.md`. Specific to this skill:

- Never change `references/voice-fingerprint.md` — that file reflects original samples, not calibration deltas.
- Never invent a rule the user didn't justify directly or by example.
- Always show the proposed style-rules diff before writing.

## Step 1 — Load workspace

1. **Locate the plugin config.** Probe these paths in order, stopping at the first hit: (1) `${EIS_CONTENT_BUILDER_CONFIG}` if set, (2) `${cwd}/.claude/eis-content-builder.local.md`, (3) walk-up from `${cwd}` checking each parent's `.claude/eis-content-builder.local.md`, (4) `${HOME}/.claude/eis-content-builder.local.md`. Use a single `Bash`:

   ```bash
   config=""
   if [ -n "$EIS_CONTENT_BUILDER_CONFIG" ] && [ -f "$EIS_CONTENT_BUILDER_CONFIG" ]; then
     config="$EIS_CONTENT_BUILDER_CONFIG"
   fi
   if [ -z "$config" ]; then
     dir="$PWD"
     while [ "$dir" != "/" ]; do
       [ -f "$dir/.claude/eis-content-builder.local.md" ] && config="$dir/.claude/eis-content-builder.local.md" && break
       dir="$(dirname "$dir")"
     done
   fi
   [ -z "$config" ] && [ -f "$HOME/.claude/eis-content-builder.local.md" ] && config="$HOME/.claude/eis-content-builder.local.md"
   echo "$config"
   ```

   Empty result → "Run `/writer-setup` first." Stop. Otherwise `Read "$config"`.
2. Extract `workspace_path`.
3. `Read` `{workspace}/references/style-rules.md`, `{workspace}/references/style-examples.md`, and (if present) `{workspace}/references/opinion-map.md`. Missing style-rules or style-examples → route to `/writer-setup` update mode. Opinion-map is optional.

## Step 2 — Detect input mode

Parse `$ARGUMENTS` and the user's message:

- Two paths given (plugin draft + user rewrite) → **Mode A: diff mode**
- `--describe` flag OR only a prose description → **Mode C: describe mode**
- One path given, no rewrite → ask: "Do you have a rewritten version of this piece, or do you want to just tell me what was off?" — resolve to A or C.
- Nothing given → ask: "Calibration works two ways. Paste your rewritten version, or just tell me what didn't sound right. Which?"

## Step 3A — Diff mode

1. **Identify both files.**
   - Draft path: must be inside `{workspace}/drafts/` OR a final saved path under the workspace structure. `Read` it.
   - Rewrite path: can be anywhere readable. `Read` it.

2. **Compute structural diff.** Not a char-by-char diff — a **voice diff**. Compare:

   - **Openings** — is the rewrite's first paragraph substantially different? If yes, extract both. This becomes a new example pair.
   - **Closings** — same.
   - **Removed phrases** — phrases present in draft, absent in rewrite. Candidates for the forbidden list.
   - **Added phrases** — phrases introduced by the rewrite that appear in other sample-sources (check via `Glob` + `Read` on `sample-sources/`). If a phrase appears in ≥ 2 samples or is otherwise distinctive, it's a signature phrase to add to preferred expressions.
   - **Paragraph rhythm delta** — did short paragraphs get merged? Long paragraphs split? Note direction.
   - **List-to-prose conversion** — did lists become prose or vice versa? Note direction.
   - **Link density** — more/fewer links? Note.
   - **Tone shift markers** — did formal language become casual? Jargon removed? Slang added?
   - **Position shift (opinion-map signal)** — did the rewrite *change what the piece claims*, not just how it claims it? If a categorical claim was softened, a firm position was hedged, a hedged position was firmed up, or a previously-unmentioned stance was added — flag it as an opinion-map signal. Do **not** auto-write to opinion-map.md. Surface it as a question to the user at the end of the calibration diff.

3. **Synthesize proposed rule deltas.** For each finding, propose a concrete rule change:

   - Removed phrases → add to `## Forbidden Phrases and Patterns` with a short `why: spotted in draft, removed in rewrite` note.
   - Added phrases → add to `## Preferred Expressions and Constructions`.
   - Opening/closing changes → add a new real example in `style-examples.md` (✅ Real from rewrite, ❌ Generic from draft).
   - Paragraph rhythm shift → update the `## Sentence and Paragraph Patterns` prose with a sentence like "Prefer merging paragraphs under 8 words into the preceding one."
   - List/prose shift → update the `## Formatting Rules → Lists` line.
   - Tone shift → update the voice principle that most matches; if none fits, propose a new voice principle.

4. **Show the proposed diff.** Format:

   ```markdown
   ## Proposed calibration

   ### style-rules.md changes
   - Forbidden: + "{phrase}" (removed in your rewrite)
   - Preferred: + "{phrase}" (added in your rewrite, matches {n} other samples)
   - Formatting > Lists: {old line} → {new line}

   ### style-examples.md additions
   New opening example pair:
   - ✅ Real (from your rewrite): "{quote}"
   - ❌ Generic (from plugin draft): "{quote}"

   ### Not changing
   - voice-fingerprint.md (regenerated only by /writer-setup)
   ```

5. **Ask confirmation.** `AskUserQuestion`:

   - "Apply all"
   - "Apply some (I'll tell you which)"
   - "Drop all — I was just thinking out loud"

   If "apply some", let the user say which bullets to keep. Apply those only.

6. **Apply.** Use `Edit` (not `Write`) to patch `style-rules.md` and `style-examples.md`. One `Edit` per logical change so the diff stays reviewable.

7. **Confirm.** Tell the user what changed, in which file, and suggest: "Next time you run `/writer-create`, these rules are live."

8. **Update opinion-map (if signals detected).** If the analysis in step 2 flagged a position shift, invoke the `opinion-extractor` agent in `single` mode against the user's rewrite. Pass:
   - `mode`: `single`
   - `source_paths`: [path to the user's rewrite, or a temp file containing the rewrite text]
   - `workspace_path`: `{workspace}`
   - `existing_opinion_map_path`: `{workspace}/references/opinion-map.md`
   - `verbose`: false in silent mode, true in interactive mode (default for this skill is interactive — calibrate is conversational by nature)

   For each `New positions` entry from the agent:
   - **Silent path** (when calibrate is invoked from another skill non-interactively): apply the `Apply patch` directly via `Edit` and append the changelog line with trigger `writer-calibrate`. Cap confidence at `média` (single-source rule).
   - **Interactive path** (default for `/writer-calibrate`): show the user the proposed addition and ask:

     > "Sua reescrita mudou a posição do texto sobre {tema}: {old} → {new}. Quer registrar em `opinion-map.md` agora (confiança: média), abrir `/writer-opinion-mine` para conversar mais a fundo, ou deixar para depois?"

   For `Conflicts` entries (rewrite contradicts a Núcleo duro position): never auto-apply. Surface to the user with the recommendation to run `/writer-opinion-mine {tema}`.

## Step 3C — Describe mode

1. **Ask targeted questions.** Use `AskUserQuestion` — one question per category, only ask the ones that are relevant based on what the user said:

   - "What part was off?" → Options: "Opening", "Body flow", "Tone", "Specific phrases", "Formatting (lists/bold/links)", "Closing"
   - Follow with a free-text: "In your words — what would have been right?"

2. **Translate prose into rule deltas.** This is the risky part. Be literal, not creative.

   - If the user says "too formal" → **don't** invent a register change. Ask: "Can you give me one sentence from the draft that was too formal, and how you'd say it?" Then treat that as a diff-mode input (mini-Mode A).
   - If the user says "used too many lists" → update `Formatting Rules → Lists` with: "Avoid lists in {channel identified} pieces. Use prose unless the content is genuinely enumerable."
   - If the user says "opening was generic" → ask them to write or paste a better opening for the same topic. Then add an example pair to `style-examples.md`.
   - If the user says "this phrase sucks" → add to forbidden list with the phrase verbatim.

3. **Never commit a vague rule.** If the user's feedback cannot be translated into a concrete rule change (phrase, example, or specific formatting rule), tell them: "I can't encode that as a rule without an example. Can you show me one place where it happened and how you'd fix it?"

4. **Show proposed diff, confirm, apply.** Same as Mode A steps 4–6.

## Step 4 — Close

Tell the user:

- How many rules changed
- How many new examples were added
- Whether voice-fingerprint would benefit from a fresh run (suggest `/writer-setup` update mode if the user has added 3+ new rules since the last fingerprint regeneration — track this heuristically by checking `references/voice-fingerprint.md` meta date vs changes)

## TodoWrite usage

Use for Mode A when there are 4+ proposed changes. One todo per change cluster (openings, closings, forbidden, preferred, formatting). Do not use for Mode C unless it escalates into multiple diffs.

## Failure handling

- User gives only vague description, cannot produce an example → refuse gracefully: "Calibration needs evidence — a phrase, a paragraph, or a rewrite. Otherwise I'd be guessing."
- Draft or rewrite file unreadable → ask for paste-in-chat fallback.
- The rewrite is drastically different topic from the draft → stop: "These look like different pieces. Calibration needs the draft + your rewrite of the same draft."
